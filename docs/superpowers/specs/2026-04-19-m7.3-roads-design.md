# M7.3 — Roads (Maze + Midpoint Displacement)

**Status:** Spec
**Date:** 2026-04-19
**Predecessor:** M7.2 (Trees, closed)
**Successor:** M7.4 (Buildings — extrude from road pixels)
**Reference:** `docs/procedural-generation-vq.md` §2 (Road Networks)

## Goal

Port VoxelQuest's road network generation: inter-city paths via recursive midpoint displacement with terrain-weighted cost, intra-city streets via recursive backtracking maze with height-delta tiebreakers. Roads are written as pixel trails into a per-block raster, sampled by `generate.comp.glsl` at voxel-generation time, and override terrain with a road material. First system in M7 that persists per-block data beyond the heightmap.

## Scope

### In

- Per-`Block` road raster: `R8UI` texture, `BLOCK_HEIGHTMAP_PIXELS²` (= 512²) pixels. One byte per world pixel with bitflags (`ROAD_BIT = 0x01`; room reserved for future lane types).
- CPU-side `RoadRaster` (`char[]`) that the generation algorithms write into, then upload to the GL texture alongside the heightmap.
- **Sparse city placement** via `city_hash(cx, cz) % CITY_SPARSITY == 0` on both axes (`CITY_SPARSITY = 4` → 1/16 blocks). Cities land at block centers. Stable per-block (no recompute on neighbor changes).
- **Intra-city maze** — recursive backtracking on a 16×16 grid covering a 128×128 world-pixel footprint (`MAZE_CELL_PX = 8`), height-delta tiebreaker for neighbor choice. Matches VQ's `btStack` loop in `f00380_gameworld.hpp`.
- **Inter-city midpoint displacement** — recursive `best_path` with `NUM_TRIES = 16` random midpoint candidates per level, terrain-gradient-weighted cost, `WATER_PENALTY` for below-sea-level crossings. Max recursion depth 10.
- Junction handling by OR-overlap on the raster (bitwise): no snap/endpoint merging logic yet.
- `city_size_for_block(cx, cz)` helper returning 128 today; plumbing ready for hash-variable sizes later (**C infra, A behaviour**).
- Shader binding: `generate.comp.glsl` gains `usampler2D road_raster` at binding 7. Samples at `(world.xz)` normalised into block coords; surface voxels (`world.y == int(h_w)`) with the ROAD_BIT set render as `Material.DIRT`.

### Out (owned later)

- Heightmap flattening under roads (M7.4 polish or deferred — roads as flat material on the original heightmap for M7.3).
- Multi-lane distinctions (main road vs alley vs bridge).
- Shipping routes (water-reward cost function, M10).
- Junction endpoint snapping (M7.4 if buildings reveal visible parallel-road artifacts).
- Block recompute when city sparsity rule changes.
- Road-adjacency lookup API for M7.4 (we expose `raster[i]` directly; M7.4 wraps it when it needs per-pixel queries).

## Data Shapes

### `src/world/road.c3` (new)

```c3
module c3voxel::world;

import std::core::mem::alloc;

import c3voxel::core;

const uint ROAD_BIT        = 0x01u;     // surface pixel is part of any road
const int  MAZE_CELL_PX    = 8;          // 16 cells × 8 px = 128 px footprint
const int  CITY_SPARSITY   = 4;          // one city per CITY_SPARSITY² blocks
const int  INTER_NUM_TRIES = 16;         // candidate midpoints per recursion level
const int  INTER_MAX_GEN   = 10;          // ~2^10 = 1024 segments worst case
const int  WATER_PENALTY   = 64;          // cost units for a below-sea pixel
const float WATER_SEA_LEVEL_NORM = 0.25f; // h01 < this counts as water

struct RoadRaster {
    char[]     pixels;              // BLOCK_HEIGHTMAP_PIXELS² bytes
    int        pitch;                // BLOCK_HEIGHTMAP_PIXELS (redundant; convenience)
    bool       initialized;
    Allocator  allocator;
}

fn void RoadRaster.init(&self, Allocator a);
fn void RoadRaster.free(&self);
fn void RoadRaster.clear(&self);        // zeroes the byte buffer; keeps capacity

fn bool block_has_city(BlockCoord cx, BlockCoord cz);   // pure hash check
fn int  city_size_for_block(BlockCoord cx, BlockCoord cz);   // returns 128 today
fn void generate_roads_for_block(Allocator scratch, Block* block, uint world_seed);
```

`Block` gains:

```c3
struct Block {
    // ... existing fields ...
    RoadRaster     road;
    gpu::Texture2D road_texture;   // R8UI, same dims as heightmap
}
```

Lifecycle mirrors `heightmap`:
- `BlockPool.get_or_create` creates the texture + inits `road`.
- `BlockPool.decrement_ref` (zero refcount path) destroys both.
- `generate_roads_for_block` is called once per block, right after `generate_heightmap_cpu`, then uploaded via `gpu::texture2d_upload`.

### Shader UBO / binding layout

Bindings currently used by `generate.comp`:
`0` volume image, `1` heightmap, `2` U uniform, `3-5` geom/tree/line SSBOs, `6` voronoi UBO.

Add **binding 7: `usampler2D road_raster`** (R8UI, same dimensions and UV space as the heightmap).

`GenerateUniforms` unchanged — we already have `block_origin` and `block_extent`, and the raster matches the heightmap coordinate system exactly, so the same `hm_uv` computation reuses.

## Algorithms

### Sparse city selection

```c3
fn bool block_has_city(BlockCoord cx, BlockCoord cz) {
    int ix = (int)cx;
    int iz = (int)cz;
    // Modulo over a checkerboard pattern so cities appear every N blocks on each axis.
    if (((ix % CITY_SPARSITY) + CITY_SPARSITY) % CITY_SPARSITY != 0) return false;
    if (((iz % CITY_SPARSITY) + CITY_SPARSITY) % CITY_SPARSITY != 0) return false;
    return true;
}
```

### Intra-city maze

Port of VQ `f00380_gameworld.hpp` recursive backtracker, specialised to a 16×16 cell grid:

```c3
fn void generate_maze(RoadRaster* r, Block* block, int center_px, int center_pz, int footprint_px, uint seed);
```

Iterative stack-based DFS:
1. Initialise `visited[16][16] = false`.
2. Push `(8, 8)` (center cell) onto stack.
3. While stack non-empty:
   - Peek the top cell.
   - Enumerate its 4 neighbours in a **deterministically randomised order** via the same `voronoi_hash` seed pattern as plant_gen.
   - For each candidate neighbour that is unvisited AND inside bounds, compute the `|Δh|` between cell centres by sampling `block.heightmap` CPU reference (or re-deriving via `fbm2` since we don't keep CPU heightmap — TBD).
   - Pick the neighbour with the smallest Δh. Ties break by order.
   - If none valid, pop stack, continue.
   - Else: `draw_bresenham_segment(r, current_cell_center_px, neighbour_cell_center_px, ROAD_BIT)`. Mark visited. Push neighbour.

**Heightmap access from CPU:** Block only stores a GPU texture. Two options — leave the choice to the plan:

- (Preferred) `generate_heightmap_cpu` currently allocates, fills, uploads, and frees `buf`. Extend it to keep `buf` on `block` as a persistent `float[]`. Memory: `512² × 4 B = 1 MB` per block, × ~16 resident blocks ≈ 16 MB. Acceptable; also serves M7.4's building-placement needs.
- (Fallback) Sample `noise::fbm2` directly with the same parameters — matches the GPU heightmap shader and avoids the extra storage. 16 cell edges × N pixels per edge × 1 fbm2 call each ≈ cheap.

Plan step will choose; spec commits to whichever keeps M7.4 cleanest. Leaning the former since M7.4 will want it too.

### Inter-city midpoint displacement

Port of VQ `bestPath`:

```c3
fn float best_path(
    RoadRaster* r, Block* block,
    float x1, float z1, float x2, float z2,   // block-local pixel coords
    int  gen, bool do_set, uint* counter, uint seed) @local;

fn float weigh_path(Block* block, float x1, float z1, float x2, float z2) @local;
```

Algorithm:
1. If `rad < 2.0 || gen > INTER_MAX_GEN`: if `do_set`, draw a Bresenham segment p1→p2, setting `ROAD_BIT`, return 0.0.
2. Else: `mid = midpoint(p1, p2); rad = dist(p1, p2) / 2;`
3. For `i in 0..INTER_NUM_TRIES`: offset `mid` orthogonally by `(rand - 0.5) * rad` on each axis; evaluate `cost = weigh_path(p1, cand) + weigh_path(cand, p2)`; keep the lowest-cost `cand`.
4. Recurse into `(p1, best_cand, gen+1)` and `(best_cand, p2, gen+1)`, passing `do_set` through.

`weigh_path` samples heightmap along the Bresenham line at pixel granularity: sum `|h[i+1] - h[i]| + (h < sea_level ? WATER_PENALTY : 0)` over all pixels on the segment.

### Cross-block connections

The sparse-city rule means every city has exactly 8 potential neighbour cities (adjacent by `CITY_SPARSITY` on each axis). For each resident city-block, connect to its **north and east** neighbours only (asymmetric handshake: west/south handle the reverse). This:

- Avoids double-drawing the same segment from both sides.
- Produces symmetric roads when both blocks are resident, since the deterministic `best_path` returns identical pixels from either direction.
- When only one side is resident, that side draws the full segment (extending into the neighbour's raster via clamp-or-clip).

Pixel coordinates are block-local `[0, 512)`. Routes crossing block edges are **clipped** against the current block's raster — pixels outside bounds are dropped. The neighbour block regenerates independently and draws its own half.

For M7.3 we accept that cross-block roads may have a 1–2 pixel seam at the edge when terrain cost drove the midpoint one direction on one side and the opposite on the other. Post-M7.3 could share a seed-derived "boundary mid-point" to guarantee continuity.

### Bresenham segment drawing

```c3
fn void draw_segment(RoadRaster* r, int x1, int z1, int x2, int z2, char flag_bits) @local;
```

Standard Bresenham over `(dx, dz)`. Sets `r.pixels[z * pitch + x] |= flag_bits` for each plotted pixel; clips pixels outside `[0, pitch)`.

## Per-Block Lifecycle

1. **`BlockPool.get_or_create`:** allocate `road.pixels`, create `road_texture` (`R8UI`, 512×512).
2. **`main.c3::drain_dirty`, inside `if (block.heightmap_dirty)`:**
   - `generate_heightmap_cpu(mem, block, WORLD_SEED)` — existing.
   - `generate_roads_for_block(mem, block, WORLD_SEED)` — new; fills `road.pixels`, uploads to `road_texture`.
3. **`GenerateContext.dispatch_page`:** binds `road_texture` at binding 7 alongside the existing `heightmap`.
4. **`BlockPool.decrement_ref` (zero refcount):** `road.free()`, `texture2d_destroy(road_texture)`.

## GPU Evaluation

`generate.comp.glsl` gains (placed just after terrain_color computation):

```glsl
layout(binding = 7) uniform usampler2D road_raster;

// After computing h_w (terrain surface y):
uint  road_flags = texture(road_raster, hm_uv).r;
float surface_top = h_w;
if ((road_flags & 1u) != 0u && abs(world.y - surface_top) < 1.0) {
    result = material_color(MATERIAL_DIRT_ID);   // M7.3 uses Material.DIRT
}
```

Sampled at the same `hm_uv` the terrain band uses, so road pixels are aligned with heightmap pixels exactly. Surface-voxel check keeps roads on the ground plane (not extruded in Y).

## Testing

| Check                  | Expected                                                                        |
|------------------------|---------------------------------------------------------------------------------|
| Build                  | `c3c build linux` green. `$assert`s unchanged (no new UBO).                     |

| Eviction               | Evict a block, re-enter it — same road raster re-generates.                     |

## Risk Register

| Risk | Mitigation |
|------|------------|
| CPU heightmap storage doubles block memory | See §Intra-city maze. If we go CPU-persistent, document the 1 MB/block cost; cap by existing block-pool size. |
| `best_path` recursion explodes for far-apart blocks | `INTER_MAX_GEN = 10` hard-caps at ~1024 segments; Bresenham tail closes out. |
| Cross-block seam at road edge | Accepted for M7.3; documented above. |
| Maze floods water pixels | `generate_maze` skips neighbour cells where `h01 < WATER_SEA_LEVEL_NORM` — village doesn't extend into water. |
| `usampler2D` filter mode | R8UI with `GL_NEAREST` (already the default for integer formats in `texture2d_create`). |
